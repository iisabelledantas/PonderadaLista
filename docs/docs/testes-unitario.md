# Testes Unitários

Os testes unitários têm como objetivo garantir a corretude das regras de negócio da aplicação de forma isolada, validando o comportamento das principais camadas sem dependência de serviços externos.

Para garantir isolamento total, são utilizados mocks para o banco de dados e serviços da AWS, permitindo testes rápidos, determinísticos e independentes do ambiente externo.

## Ferramentas

| Ferramenta | Versão | Finalidade |
|---|---|---|
| JUnit 5 | - | Framework de testes unitários |
| Mockito | - | Criação de mocks, stubs e verificação de interações |
| mockito-kotlin | 5.2.1 | Suporte à sintaxe Kotlin no Mockito (`any()`, `verify`, etc.) |
| MockitoExtension | - | Integração JUnit 5 + Mockito via `@ExtendWith` |

## Cobertura dos Testes

### TelemetriaConsumerTest (TelemetriaProcessor)

Testa a lógica de processamento do lote de mensagens SQS e persistência em batch no banco.

| Teste | Cenário | Resultado Esperado |
|---|---|---|
| `deve processar batch com mensagem valida sem lancar excecao` | Uma mensagem JSON válida | Nenhuma exceção lançada |
| `deve executar batch insert ao processar mensagem valida` | Uma mensagem válida | `addBatch()` chamado 1x, `executeBatch()` chamado 1x |
| `deve executar batch insert para cada mensagem valida no lote` | Três mensagens válidas | `addBatch()` chamado 3x, `executeBatch()` chamado 1x |
| `deve descartar mensagem com body invalido sem lancar excecao` | Body vazio | Nenhuma exceção, `addBatch()` nunca chamado |
| `deve processar mensagens validas e descartar invalidas no mesmo batch` | 2 válidas + 1 inválida | `addBatch()` chamado 2x, batch executado normalmente |
| `deve reutilizar conexao existente em warm start` | Duas invocações consecutivas | `prepareStatement` chamado 2x, `close()` nunca chamado |
| `deve lancar excecao quando executeBatch falhar` | `executeBatch()` lança `RuntimeException` | Exceção propagada, `autoCommit = false` verificado |
| `nao deve fazer nada com lista vazia` | Lista de mensagens vazia | Nenhuma interação com a conexão |

### TelemetriaControllerTest

Testa o endpoint REST e a delegação ao `SqsService`.

| Teste | Cenário | Resultado Esperado |
|---|---|---|
| `deve retornar 200 OK ao receber dados válidos` | DTO válido recebido via POST | Status HTTP 200 |
| `deve chamar o sqsService ao receber dados válidos` | DTO válido recebido | `sqsService.sendMessage()` chamado exatamente 1x |
| `deve lancar excecao ao falhar serialização` | `ObjectMapper` lança exceção | Exceção propagada ao controller |

### SqsServiceTest

Testa o envio de mensagens para a fila SQS via AWS SDK.

| Teste | Cenário | Resultado Esperado |
|---|---|---|
| `deve chamar o sqsClient ao enviar mensagem` | Mensagem válida | `sqsClient.sendMessage()` chamado exatamente 1x |
| `deve lancar excecao quando o sqsClient falhar` | `SqsClient` lança `RuntimeException` | Exceção propagada ao chamador |
